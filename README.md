# Cinemory Mobile

> Native iOS + Android app for [Cinemory](https://github.com/upgradedev/cinemory)
> — pick photos from your device, choose an occasion, and turn them into a
> cinematic reel to share. One [Flutter](https://flutter.dev) codebase for both
> platforms.

This is the **mobile foundation** that unlocks the one thing the web app can't
do: **on-device photo-library access**. Apple Photos / iCloud has no server-side
read API — the only sanctioned path to a user's originals is a native app using
**PhotoKit** with the user's permission. This app is that path (PhotoKit on iOS,
MediaStore on Android) via the [`photo_manager`](https://pub.dev/packages/photo_manager)
plugin.

---

## What it does

```
┌──────────────┐   ┌───────────────┐   ┌──────────────┐   ┌───────────────┐
│ 1. Pick      │──▶│ 2. Occasion   │──▶│ 3. Generate  │──▶│ 4. Preview +  │
│ on-device    │   │ template      │   │ (Cinemory    │   │ native Share  │
│ photos       │   │ (6 presets)   │   │  API)        │   │ + Save        │
└──────────────┘   └───────────────┘   └──────────────┘   └───────────────┘
  PhotoKit /        anniversary /        POST /reels        share_plus →
  MediaStore        wedding / …          → reel + SHA-256   IG/FB/LinkedIn/YT
  (permission)                           provenance         + save to gallery
```

- **On-device photo access** — `photo_manager` → PhotoKit (iOS/iCloud, on-device
  with permission) and MediaStore (Android). Proper permission/consent flow,
  including iOS/Android "limited/selected photos" grants.
- **Occasion templates** — the six Cinemory presets (anniversary, graduation,
  birthday, wedding, year-in-review, business-event), fetched from
  `GET /occasions` with a built-in fallback so the picker works offline.
- **Generate** — calls the Cinemory API `POST /reels` and shows the returned
  provenance (reel name, steps, SHA-256).
- **Native share + save** — `share_plus` opens the OS share sheet (Instagram,
  Facebook, LinkedIn, YouTube — **no per-platform API review**); `gal` saves the
  reel to the device gallery.

## Architecture

```
lib/
  main.dart            app root; wires services into AppState (Provider)
  config.dart          API base URL via --dart-define; no secrets
  theme.dart           Cinemory brand (cinematic dark, gold)
  models/
    occasion.dart      GET /occasions shape + 6 fallback presets
    reel.dart          ReelRequest (POST /reels) + Reel response
    picked_photo.dart  platform-agnostic photo handle (keeps state testable)
  services/
    photo_service.dart PhotoGallery abstraction + PhotoManagerGallery (PhotoKit/MediaStore)
    cinemory_api.dart  typed API client (http) + upload seam (see gap below)
    share_service.dart download → share sheet (share_plus) + save (gal)
  state/app_state.dart ChangeNotifier driving the whole flow
  screens/             intro · gallery · occasion · generating · preview
  widgets/             photo_grid_tile · occasion_card
```

The UI and `AppState` depend only on the `PhotoGallery` interface and the typed
`CinemoryApi`, so the entire flow is unit-testable with fakes — no platform
channels, no network (see `test/`).

## ⚠️ Known gaps to flag (by design — not worked around)

1. **The Cinemory API cannot yet receive photos.** `POST /reels` generates from
   a *synthetic* spec — it takes `{name, chapters, per_chapter, occasion}` and
   has **no endpoint that accepts uploaded photo bytes**. So this app reads real
   on-device photos and drives the full pick→occasion→generate→share loop, but
   the pixels the user picks are **not sent** — the request carries the occasion
   and a chapter count *derived from how many photos were picked*. Wiring the
   real photos through needs a new multipart ingest endpoint on the API
   (`POST /reels/upload`). The client seam is already in place and marked:
   `CinemoryApi.uploadPhotosAndGenerate()` (throws `UnimplementedError` until the
   server endpoint lands).

2. **iOS builds require macOS.** Flutter compiles iOS only on macOS/Xcode. The
   founder develops on Windows and can build/run **Android** locally; **iOS is
   cloud-only** — the `build-ios` CI job runs on a `macos-latest` runner.

3. **Store submission is account-gated (user steps).** Shipping needs an **Apple
   Developer account ($99/yr)** + signing certs/provisioning + review, and a
   **Google Play Developer account ($25)**. The CI iOS build is **unsigned**
   (`--no-codesign`) — it proves compilation, not shippability. Exact steps:
   [`docs/STORE_SUBMISSION.md`](docs/STORE_SUBMISSION.md).

## Bootstrap (first run)

The repo intentionally **does not commit** the native `android/` and `ios/`
folders (they contain binary blobs — the gradle wrapper jar, the Xcode project —
and package-name config that drifts). They are generated by `flutter create`,
then patched with the photo/media permissions:

```bash
# one-time, and after any Flutter SDK change
bash tool/bootstrap.sh          # macOS / Linux / Git-Bash
pwsh tool/bootstrap.ps1         # Windows PowerShell
```

This runs `flutter create --org com.upgradedev --project-name cinemory_mobile
--platforms=android,ios` into a temp dir, copies `android/` + `ios/` in, and runs
`tool/apply_platform_overlays.py` to inject:

- **Android** (`AndroidManifest.xml`): `READ_MEDIA_IMAGES`, `READ_MEDIA_VIDEO`,
  `READ_MEDIA_VISUAL_USER_SELECTED`, `READ_EXTERNAL_STORAGE` (maxSdk 32),
  `ACCESS_MEDIA_LOCATION`, `WRITE_EXTERNAL_STORAGE` (maxSdk 29, for gallery save).
- **iOS** (`Info.plist`): `NSPhotoLibraryUsageDescription`,
  `NSPhotoLibraryAddUsageDescription`.

The reference copies live in [`platform_overlays/`](platform_overlays/).

## Run

```bash
# Android (works on Windows/Linux/macOS)
flutter run --dart-define=CINEMORY_API_BASE_URL=http://10.0.2.2:8000

# iOS (macOS only)
flutter run --dart-define=CINEMORY_API_BASE_URL=http://localhost:8000
```

`CINEMORY_API_BASE_URL` points at your running Cinemory API (see the
[cinemory repo](https://github.com/upgradedev/cinemory): `uvicorn cinemory.api:app`).
`10.0.2.2` is how the Android emulator reaches your host machine.

## Test & lint

```bash
flutter pub get
flutter analyze          # flutter_lints + project rules
flutter test             # unit + widget tests (all offline, faked)
```

Tests never touch platform channels: the photo library is faked
(`FakeGallery`) and the API is driven by `http`'s `MockClient`.

## CI (GitHub Actions — [`.github/workflows/ci.yml`](.github/workflows/ci.yml))

| Job | Runner | Proves |
|---|---|---|
| `secret-scan` | ubuntu | gitleaks v8.18.4 `--redact`, fail-fast |
| `analyze-and-test` | ubuntu | `flutter analyze` + `flutter test` |
| `build-android` | ubuntu | bootstrap + `flutter build apk --debug` → uploads APK |
| `build-ios` | **macos-latest** | bootstrap + `flutter build ios --no-codesign` |

Android builds on Linux; **iOS builds on macOS** because Flutter iOS compilation
requires Xcode. The iOS job is unsigned — signing/upload is account-gated (above).

## Notes on the toolchain

CI pins **Flutter `3.44.4`** (latest stable) for reproducibility. The plugin set
(`photo_manager`, `share_plus` 13.x with the `SharePlus.instance` API,
`video_player`, `gal`, `path_provider`) is aligned to that toolchain — the newest
plugin Android build files require the matching Flutter Gradle tooling, so an
older pinned Flutter (e.g. 3.24) fails the Android build with
`unknown property 'flutter'`. Bump both together.
